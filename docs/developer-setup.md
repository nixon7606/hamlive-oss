# Developer Setup

This guide covers environment setup, build processes, and development workflows for Ham.Live.

## Quick Start

**Prerequisites:** Node.js and MongoDB installed locally.

1. **Install dependencies:**

```bash
npm install
```

2. **Start development with auto-restart and TypeScript watching:**

```bash
npm run dev
```

3. **Open your browser:** http://localhost:3000

> **Note:** The development server runs on port 3000 by default. You can access the Ham.Live application at `http://localhost:3000` and API endpoints at `http://localhost:3000/api/*`.

## Environment Configuration

Ham.Live uses a layered configuration system with YAML configuration files and environment variables.

### How configuration is loaded

`server/dist/lib/configLib.js` loads configuration in this order:

1. Root `.env` file (one unconditional path: `../../../.env` relative to `server/dist/lib/`) — create it by running `npm run setup` or copying `.env.example` to `.env`. This is the only `.env` file the application reads.
2. Base YAML: `server/dist/commonConfig.yaml`
3. Environment-specific YAML: `server/dist/devConfig.yaml` (when `NODE_ENV=development`) or `server/dist/prodConfig.yaml` (otherwise)
4. Environment variables from the process: override any YAML value

> **Note:** `server/dist/.env-development` and `server/dist/.env-production` are inert stub files — they are **not loaded by the application**. All live configuration belongs in the root `.env` (or in real environment variables). Do not rely on or edit those stub files.

**See [Runtime Configuration](runtime-config.md) for complete configuration documentation.**

## Build Process

### Development Build

```bash
npm run dev          # Start development server (auto-creates .env, starts MongoDB, watches TS)
npm run build        # One-shot TypeScript build (client + server)
npm run dev:watch    # Runs the watchers and nodemon directly (called internally by npm run dev)
```

### Production Build

```bash
npm run build        # Compile TypeScript sources
npm start            # Start production server (requires a prior npm run build)
```

### Client Assets

- **Source:** `client/src/public/js/`
- **Output:** `client/dist/public/js/`
- **Pattern:** Per-view entry points (`byView/*/main.ts`) compiled to ES modules

### Server Assets

The project is **partway through a JavaScript → TypeScript migration**:

- **Four server-side TypeScript files** live under `server/src/lib/`: `realtimeClients.ts`, `responseUtils.ts`, `secureSign.ts`, and `streamChat.ts`. Edit those in `server/src/` and let `npm run build` (or the watcher) regenerate their `server/dist/` counterparts.
- **Everything else in `server/dist/`** (controllers, models, routes, most of lib, EJS views, CLI tools) is hand-maintained JavaScript that has **no separate TypeScript source**. Edit those files directly in `server/dist/` — that JavaScript **is** the source of truth.
- **Never delete `server/dist/`** — it is not a generated output directory in the usual sense. Only `client/dist/` is fully regenerated by the build.

> **Runtime:** Node.js + Express with EJS templates, running directly from `server/dist/`.

## Development Workflow

### TypeScript Compilation

The project uses TypeScript with ES modules (no bundling):

- Client and server TypeScript compile separately
- Per-view entry scripts maintain runtime ES module loading
- Shared libraries compile to `client/dist/public/js/lib/`

### Syncing shared types

The shared type definitions are maintained in the client tree and copied to the server tree as needed:

```bash
npm run sync:types   # Copy client/src/public/js/types/commonTypesupport.ts → server/src/types/
```

Run this after editing the shared types file so the server TypeScript picks up the changes.

### Code quality

Formatting is handled by Prettier (`.prettierrc.json`). Run your editor's Prettier integration or `npx prettier --write` on files you touch. ESLint config lives in `client/.eslintrc.cjs` (for client code).

## Development Tips

### Environment Variables

```bash
# Check current environment
echo "NODE_ENV: $NODE_ENV"
echo "PORT: $PORT"
echo "LOG_LEVEL: $LOG_LEVEL"

# Debug configuration loading
node -e "const { conf } = require('./server/dist/lib/configLib.js'); console.log(JSON.stringify(conf, null, 2));"
```

### Database Setup

The application connects to whichever database is named in the `MONGODB_URI` environment variable. The default in `.env.example` (and therefore in the auto-created `.env`) is:

```
MONGODB_URI=mongodb://localhost:27017/hamlive?directConnection=true
```

The database name is **`hamlive`**. (The `dbname: hamlive-dev` field in `server/dist/devConfig.yaml` is used only by `server/dist/bin/dbBackup.js` when naming backup files — it does not affect the connection.)

`npm run dev` starts a local MongoDB automatically. If you want to run it in its own terminal (for example, to keep data across app restarts), use the bundled helper — it starts an in-memory single-node replica set and prints the URI:

```bash
npm run mongo:dev    # terminal 1 — leave running (data is in-memory; lost when this stops)
npm run dev          # terminal 2 — the app
```

For a persistent store, use Docker instead:

```bash
docker compose up -d    # starts MongoDB with a named volume (data persists across restarts)
```

To verify a running instance with the shell:

```bash
mongosh mongodb://localhost:27017/hamlive
```

### Hot Reloading

The development server (`npm run dev`) provides:

- **Server:** Auto-restart on `.js` file changes via nodemon (2.5 s debounce)
- **Client TypeScript:** Watch compilation — edits to `client/src/` are compiled to `client/dist/` automatically
- **Server TypeScript (the four `server/src/` files):** Also watched and compiled automatically
- **Templates:** EJS templates are read from disk on each request, so edits take effect immediately

## Migration Guide

Goal: migrate a per-view plain JS `main.js` to TypeScript `main.ts` while keeping no-bundler runtime.

### Checklist

1. Convert `main.js` → `main.ts` and keep exports minimal. Avoid top-level side-effects
2. Export `async function bootstrapLiveNet(deps?: { fetch?, EventSource?, clients? })` that encapsulates current top-level logic
3. In EJS views, continue loading the compiled `main.js` (output from TypeScript compiler). Use per-file compilation to ESM
4. For widgets and stores, keep the same runtime API (custom element names and store contract). Tests will import `main.ts` and call `bootstrapLiveNet` with fakes

### Migration Notes

- **DI:** pass `fetch` and `EventSource` factories in `deps` for easy mocking
- **CSS/style loading:** keep behavior unchanged; prefer to keep `fetch('/css/...')` calls isolated behind a loader helper to stub in tests

## Troubleshooting

### Common Issues

**Port conflicts:**

```bash
# Kill process using port 3000
lsof -ti:3000 | xargs kill -9
```

**TypeScript compilation errors:**

```bash
# Clean the generated client output and rebuild
rm -rf client/dist
npm run build
```

> **Important:** Do **not** delete `server/dist/` — the majority of the server source lives there as hand-maintained JavaScript. Only `client/dist/` and the four compiled server modules (`server/dist/lib/{realtimeClients,responseUtils,secureSign,streamChat}.js`) are safe to regenerate.

**Configuration issues:**

```bash
# Verify YAML config files exist
ls -la server/dist/*Config.yaml

# Test that the root .env is found and configuration loads
node -e "const { conf } = require('./server/dist/lib/configLib.js'); console.log('PORT:', conf.port || process.env.PORT, '  NODE_ENV:', process.env.NODE_ENV);"
```
